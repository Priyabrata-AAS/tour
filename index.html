Stage 02
Inventory Engine
How medicine lives inside the system. Batch tracking, expiry logic, reorder alerts.
🗂️
Batch-Driven Structure
Normal POS: Product → Quantity. Pharmacy ERP: Product → Batch → Expiry → Quantity. This is the fundamental difference.
INVENTORY TREE Dolo 650 (product_id: D123) ├─ Batch A-12 │ batch_id : B777 │ expiry : 23-05-2027 │ total_qty : 50 │ available : 43 ← total - reserved - damaged │ reserved : 5 ← pending orders │ damaged : 2 ← marked damaged │ └─ Batch C-19 (second purchase arrived) batch_id : B778 expiry : 15-08-2027 total_qty : 100 available : 100 Total Stock = 150 strips Total Available = 143 strips
🔢
FIFO Logic — Earliest Expiry First
System always sells the batch closest to expiry. Chemist does not decide. System decides automatically.
AUTO BATCH SELECTION QUERY SELECT batch_id, batch_no, expiry, available_qty FROM inventory_batch WHERE product_id = 'D123' AND available_qty > 0 AND status != 'EXPIRED' ORDER BY expiry ASC ← FIFO LIMIT 1 Result: Batch A-12 Exp May 2027 Qty 43 ← auto selected Batch C-19 Exp Aug 2027 Qty 100 ← skipped for now
⏰
Expiry Tracking Engine — Background Job
Runs every night at midnight. Scans entire inventory. Flags and alerts automatically.
CRON JOB (runs 00:00 daily) FOR EACH batch IN inventory_batch: IF expiry < today status = "EXPIRED" available_qty = 0 FIRE alert: REMOVE NOW ELSE IF expiry < today + 30 status = "CRITICAL" FIRE alert: URGENT ELSE IF expiry < today + 90 status = "NEAR_EXPIRY" FIRE alert: WARNING
⛔ EXPIRED — Remove immediately
Dolo 650 Batch A-01 3 days ago Qty: 12 ₹276 at risk
⚠ CRITICAL — Expires in 18 days
Crocin 500 Batch B-11 18 days Qty: 30 ₹600 at risk
ℹ NEAR EXPIRY — 67 days remaining
Azee 500 Batch C-22 67 days Qty: 50
📉
Reorder Level Engine
Each medicine has a minimum stock threshold. System alerts when stock drops below it.
REORDER CHECK LOGIC FOR EACH product IN products: current_stock = SUM(available_qty) FROM inventory_batch WHERE product_id = product.id IF current_stock < product.min_stock CREATE reorder_alert { product : product.name, current_qty : current_stock, min_stock : product.min_stock, order_qty : product.reorder_qty } ───────────────────────────────── 📦 REORDER ALERTS Dolo 650 Stock: 8 Min: 20 → Order 100 Azee 500 Stock: 3 Min: 10 → Order 50

Deep Dive — Barcode
Why Barcode Works.
Where It Doesn't.
The clearest possible breakdown of when barcode is your biggest advantage — and when it fails.
✅ Where Barcode WINS
Speed at billing counter — Scan takes 0.5 seconds. Typing takes 5–10 seconds. 1000 bills/day = hours saved.
Zero typo errors — "Azithromycin" typed wrong = wrong medicine sold. Barcode scan = always exact match.
GS1-128 auto-fills batch + expiry — Modern medicine boxes already have GS1-128 barcodes. One scan fills product, batch, AND expiry.
Purchase entry becomes fast — Receiving 50 medicines? Scan each box. No manual entry. Entire purchase done in minutes.
Single indexed DB query — WHERE barcode = ? uses index. Returns in under 1ms regardless of database size.
❌ Where Barcode FAILS
Old stock has no barcode — Many generic medicines and local brands come without any barcode. System must fallback to text search.
Damaged/faded barcodes — Strips that are torn, wet, or faded won't scan. System needs a manual fallback always.
Basic EAN-13 has no batch data — Most retail barcodes are EAN-13 (13 digits, product only). No batch, no expiry. GS1-128 adoption is still incomplete in Indian pharma.
Scanner hardware dependency — Barcode scanning needs physical scanner or camera. In small pharmacies without hardware, it's a barrier.
Strip-level tracking impossible — When a customer returns 1 strip from a box of 10, there is no strip-level barcode. Batch level is the finest granularity possible.
🔲
GS1-128 Barcode — What's Inside
This is the industry standard barcode on modern medicine packaging. One scan gives you everything.
0 1 8901234567890 10 A-12 17 270523
GS1-128 Standard
APPLICATION IDENTIFIER 01
8901234567890
GTIN — Global Trade Item Number. Identifies the product. Maps to product_id in your DB.
APPLICATION IDENTIFIER 10
A-12
Batch / Lot Number. Auto-fills batch field in purchase entry. Eliminates manual typing.
APPLICATION IDENTIFIER 17
270523
Expiry Date (YYMMDD format). = May 2027. Auto-fills expiry. Zero human error possible.
APPLICATION IDENTIFIER 21
SN-9981234
Serial Number (for high-value/controlled drugs). Enables unit-level tracking.
Scenario ❌ Without Barcode (Marg Style) ✅ With GS1-128 (Your System)
Receiving 50 medicines in purchase Type each name, batch, expiry manually. 50 × 7 fields = 350 inputs Scan each box. 50 × 1 scan = 50 inputs. Batch + expiry auto-filled
Billing 1 medicine Type 4 letters → select from dropdown → confirm batch Scan barcode → done. 0.5 seconds total
Wrong medicine dispensed Human typed "Amoxyclav" instead of "Amoxicillin". Possible. Barcode scan = exact product. Impossible to pick wrong medicine.
Expiry tracking during purchase Pharmacist types expiry. Typo = wrong expiry in system GS1 scan reads expiry from manufacturer. 100% accurate.
Medicine with no barcode ✓ Works fine. Text search handles it. ✗ Must fallback to text search. Scanner useless here.
Damaged/torn strip ✓ Still type name manually. ✗ Can't scan. Needs manual fallback.
Small pharmacy, no scanner ✓ Works on any computer. ✗ Needs hardware investment.
💡
The Right Answer: Hybrid System
Never force one method. Build both. Let the system decide.
SMART SEARCH LOGIC IF input = barcode_scan → WHERE barcode = ? ← fastest path ELSE IF input = text AND top medicines cache warm → Redis cache lookup ← < 1ms ELSE IF input = text → WHERE name LIKE ? ← indexed PG query IF NOTHING FOUND → Meilisearch fuzzy ← handles typos → "Did you mean Azithromycin?" All 4 paths. Zero failure mode.
